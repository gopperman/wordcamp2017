Why?

WordPress is a fantastic platform for managing content...and content producers love it

It does what it does extremely well...

  • Content Management
  • Custom Fields
  • Taxonomy
  • User Authentication
As an advanced developer, it leaves me wanting for many things. Developing a WordPress theme can be tedious. Writing disparate plugins and theme functions can create a chaotic architecture. Many great theme frameworks exist, like Sage and Timber, but those are end-runs around WordPress' limitations. If I'm coding practically from scratch, why not choose my own tools?

Use the WP REST API if:

  • You need to create a rich, interactive experience
  • Your website is action-oriented and stateful, not purely content-driven
  • You'd like to release your app on multiple platforms
  • Most of your website is a basic blog or website, except...
For example, you might be making a basic blog for a music venue, except they need a ticket reservation system

Architectural Approaches

Option 1: Integrate your app directly in a theme

Pros:

  • Take advantage of WordPress' theme features, a la carte
  • Can hybridize standard content-driven pages with more interactive experiences

Cons:

  • Not appropriate for large, complex apps
  • Who's responsible for what? (e.g. routing)

Option 2: Create a Standalone App

Pros:

  • Loosely Coupled
  • Isomorphic rendering and other optimizations
  • Host your app anywhere

Cons:

  • Potential cross-origin and authentication problems
- Theme is responsible for defining information architecture and providing content - App is responsible for rendering content - You can swap out either your javascript of your CMS for a new platform, no problem - Developers can work on the API and the app in parallel - Host your app anyway - Render your app server-side on the first load, then update using ajax - There are reasonable solutions for cross-origin security and authentication
